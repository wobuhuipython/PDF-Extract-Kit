# 分布式锁机制说明

## 问题背景

当多个进程同时处理 OSS 文件夹时，会出现以下问题：

### 问题 1: 简单的数据库检查不够

```
进程 1: 检查数据库 → 没有记录 → 开始处理
进程 2: 检查数据库 → 没有记录 → 开始处理  ← 重复处理！
```

### 问题 2: 处理周期长，数据延迟上传

```
进程 1: 获取锁 → 下载 → 提取图表 → AI 分析（耗时长）→ 上传数据
进程 2: 在进程 1 上传数据前检查 → 没有记录 → 尝试获取锁
```

## 解决方案：三重检查机制

使用 NocoDB 作为分布式锁存储，配合三重检查机制。

### 三重检查流程

```
【检查 1】查询正常记录 → 已存在？跳过
    ↓ 不存在
【检查 2】尝试获取锁 → 锁被占用？跳过
    ↓ 获取成功
【检查 3】再次查询正常记录 → 已存在？释放锁并跳过
    ↓ 仍不存在
开始处理
```

### 为什么需要三重检查？

1. **第一次检查**：快速过滤已处理的文件
2. **获取锁**：防止并发处理
3. **第二次检查**：防止在获取锁期间，其他进程已完成处理

### 实际场景

```
时间线：
T1: 进程 1 检查 → 没有记录
T2: 进程 2 检查 → 没有记录
T3: 进程 1 获取锁成功
T4: 进程 2 尝试获取锁 → 失败（被占用）→ 跳过 ✓
T5: 进程 1 处理完成 → 上传数据
T6: 进程 3 检查 → 有记录 → 跳过 ✓
```

### 锁机制详解

1. **锁记录格式**
   - 使用特殊的 `source_file` 值：`__LOCK__文件名.pdf`
   - 使用 `analysis_time` 记录锁创建时间
   - 使用特殊标记 `chart_title: __PROCESSING_LOCK__`

2. **获取锁流程**
   ```
   1. 检查是否已处理完成（有正常记录）→ 跳过
   2. 查询是否有锁记录
   3. 如果有锁：
      - 检查锁是否超时（默认 30 分钟）
      - 超时则删除旧锁，继续获取
      - 未超时则返回失败
   4. 如果没有锁：
      - 尝试插入锁记录
      - 成功则获取锁
      - 失败则说明其他进程抢先了
   ```

3. **释放锁流程**
   ```
   处理完成 → 删除锁记录
   处理失败 → 删除锁记录
   异常退出 → 删除锁记录
   ```

4. **锁超时机制**
   - 默认超时时间：30 分钟
   - 如果进程异常退出未释放锁，30 分钟后自动失效
   - 其他进程可以接管处理

## 使用示例

### 场景 1: 多进程并发处理

```bash
# 终端 1
python main.py --oss-folder public/reports/

# 终端 2（同时运行）
python main.py --oss-folder public/reports/

# 终端 3（同时运行）
python main.py --oss-folder public/reports/
```

**结果**：
- 每个文件只会被一个进程处理
- 其他进程会跳过正在处理的文件
- 不会出现重复处理

### 场景 2: 进程异常退出

```bash
# 进程 1 开始处理 file1.pdf
python main.py --pdf oss://file1.pdf

# 进程 1 异常退出（Ctrl+C 或崩溃）
# 锁记录仍然存在

# 30 分钟后，进程 2 可以接管
python main.py --pdf oss://file1.pdf
# 检测到锁超时，删除旧锁，重新处理
```

## 注意事项

1. **锁超时时间**
   - 默认 30 分钟，可以根据实际处理时间调整
   - 如果文件很大，可能需要更长的超时时间

2. **数据库要求**
   - 必须配置 NocoDB
   - 如果没有配置，锁机制不生效（允许所有处理）

3. **异常处理**
   - 程序会在 finally 块中释放锁
   - 即使异常退出，锁也会被释放
   - 如果进程被强制终止（kill -9），锁会超时后自动失效

4. **性能影响**
   - 每个文件处理前需要 2-3 次数据库查询
   - 对于大批量处理，影响很小
   - 锁操作通常在 100ms 以内完成

## 故障排查

### 问题 1: 所有文件都被跳过

**原因**：可能有残留的锁记录

**解决**：
```sql
-- 在 NocoDB 中查询锁记录
SELECT * FROM table WHERE source_file LIKE '__LOCK__%'

-- 删除所有锁记录
DELETE FROM table WHERE source_file LIKE '__LOCK__%'
```

### 问题 2: 锁一直无法获取

**原因**：其他进程正在处理，或锁未超时

**解决**：
- 等待其他进程完成
- 或等待锁超时（30 分钟）
- 或手动删除锁记录

### 问题 3: 仍然出现重复处理

**原因**：可能是数据库配置问题

**检查**：
1. 确认 NocoDB 配置正确
2. 查看日志中是否有锁相关的错误
3. 检查数据库连接是否正常

## 性能优化建议

1. **批量处理时**
   - 使用边扫描边处理模式
   - 不需要预先扫描所有文件

2. **多进程部署**
   - 可以在不同服务器上运行
   - 通过数据库自动协调

3. **监控和告警**
   - 定期检查锁记录数量
   - 如果锁记录过多，可能有进程异常
